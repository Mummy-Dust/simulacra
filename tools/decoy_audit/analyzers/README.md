# Temporal Realism Analyzer

The `rotation_audit.py` script validates the synthetic BLE device generator's temporal behavior against spec §7 requirements. It scores the synthetic crowd on four metrics of realism and optionally compares presence-duration distributions against a real capture.

## Four Temporal Checks

Each check emits PASS or FAIL and reports specific violations if found:

1. **rotation-cadence**: Static devices never rotate; RPA devices rotate at intervals strictly within the 10–20 minute band (600–1200 ms ± 1000 ms tolerance).
   - PASS: No static device has a `rotate` event; all RPA rotation gaps fall within [599, 1201] seconds.
   - FAIL: Static device rotated, or RPA gap outside band.

2. **lifetime-cohort**: Within each slot, the birth-time span of all devices is bounded by role.
   - PASS: Transient devices birth-span ≤ 720 seconds; resident devices birth-span ≤ 5400 seconds (with 1000 ms tolerance).
   - FAIL: Any role exceeds its lifetime cap.

3. **rotation-independence**: Rotation events must not synchronize (no coordinated "volley" across multiple slots).
   - PASS: No timestamp sees ≥ max(2, live_slots/2) simultaneous rotations.
   - FAIL: Synchronized rotation volley detected.

4. **no-resurrection**: Addresses are never reused; once an address leaves, a new device gets a fresh address.
   - PASS: No address appears twice across all device events.
   - FAIL: Address reused.

## Usage

### Basic validation
```bash
tools\decoy_audit\synth_dump.exe --devices <seed> <n> <ticks> <tick_ms> > run.txt
python tools\decoy_audit\analyzers\rotation_audit.py run.txt
```

The command above runs the synthetic device generator with:
- `<seed>`: Random seed (e.g., 1)
- `<n>`: Number of devices (e.g., 24)
- `<ticks>`: Simulation steps (e.g., 8000)
- `<tick_ms>`: Milliseconds per step (e.g., 1000)

Outputs four PASS/FAIL lines and `ALL TEMPORAL CHECKS PASS` on success (exit 0), or `FAILS: <details>` on violation (exit 1).

### Optional: real-vs-synth presence-duration comparison
```bash
python tools\decoy_audit\analyzers\rotation_audit.py run.txt --profile profile.json
```

If the `profile.json` was generated by `capture_profile.py` from a real capture and contains `presence_ms_bins`, the analyzer prints the Jensen-Shannon divergence between the synthetic and real presence-duration histograms. This measures how well the rotation + lifetime generator matches observed device lifespan distributions.

## `rpa_presence.py` — attacker's-eye RPA-on-air check

`rotation_audit.py` scores the *synthetic* generator. `rpa_presence.py` scores a *real capture*: it confirms the `churn_adv` raw-HCI fix actually put RPA-shaped (top-2-bits `01` / `0x40`) addresses on air. NimBLE's `ble_gap_ext_adv_set_addr` rejects that subtype, so before the fix the decoy fleet emitted **zero** RPA addresses (a third of the roster was silently dark); this check verifies the on-air reality now matches intent — the exact gap the snapshot audit could not see (it measured *intended* identities, not the air).

```bash
python tools\decoy_audit\analyzers\rpa_presence.py <capture.pcap> [min_rpa_advertisers]
# default floor = 3 distinct RPA advertisers; exit 0 = PASS, 1 = FAIL
```

It reuses `capture_profile.py`'s parser (DLT256 + Nordic DLT157) and reports distinct advertisers by address subtype (static / RPA / NRPA-or-public).

**Honest scope:** it counts *all* RPA advertisers in range — ambient real phones emit RPA too, so a nonzero count alone is not proof the decoys are the source. The decisive signal is **before-vs-after**: with the fleet running and co-located, decoys contributed zero RPA before the fix, so a healthy count now confirms they transmit it. For strict decoy isolation, cross-reference strong-RSSI / co-located advertisers or decoy archetypes. Like the rest of this suite it does **not** measure the physical layer.

## Privacy & Scope

**IMPORTANT: Captures and their profiles are never committed.** Store them only in the gitignored `private/` folder. The analyzers themselves emit no addresses, timestamps, names, or packet payloads — only pass/fail verdicts and aggregated statistics.

The temporal analyzer scores the **synthetic generator's temporal behavior** (rotation cadence, lifetime cohort membership, rotation independence, address lifecycle); `rpa_presence.py` scores **on-air subtype presence** from a real capture. Neither measures the physical layer — RSSI, angle-of-arrival, or signal fingerprinting are out of scope, consistent with the spec's honest ceiling.
